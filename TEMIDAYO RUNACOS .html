RUNACOS Web Application - Business Requirements Document (BRD)

1. Project Overview

RUNACOS is a professional academic website designed to provide institutional information, news, events, articles/blogs, and a Fault Reporting & Management System (FRMS) for public users. Public users cannot log in. Staff and administrators have access to the FRMS dashboard.

Objectives: - Provide a structured academic website for public users - Enable public users to report and track faults via FRMS - Maintain administrative control over reported faults - Present academic content such as news, articles, and events clearly and professionally

3. Functional Requirements

3.1 Public Website Pages

  • Home: Hero section, latest news, upcoming events, featured articles, quick access cards (Report/Track Fault, View Events, Latest News)
  • About RUNACOS: Overview, Vision, Mission, Core Values, Objectives
  • News & Updates: List of news articles with details page
  • Events / Workshops / Seminars: List of events (upcoming/past), event details page, optional registration modal popup
  • Articles / Blogs: Article list and details pages
  • Contact / Stay in Touch: Contact form, map, address, phone, email
  • Privacy Policy / Terms of Service: Informational pages
  • 404 / Error Page: Custom error page with home link

3.2 Fault Reporting & Management System (FRMS)

Public Access: - Report a Fault form with fields: Name, Email, Phone, Location, Category, Description, File Upload - Track My Fault with Reference ID, showing status, assigned department, date submitted, last update

Admin/Staff Access: - Dashboard with KPI cards (Total, Open, In Progress, Resolved) - Table of reported faults - Filters, search, status update, staff assignment - Fault category management - Reports generation - User management (Admin Only)

Notes: - Fault submissions are recorded upon user submission (no automated logging or portal access) - Reference IDs generated for tracking purposes only

3.5 Past Questions (PQ) Upload & Download Module

Purpose:

Allow administrators to upload past questions (PQs) for public access. Public users can browse and download PQs without logging in.

Admin Functionality:

  • Upload PQ files via admin dashboard
  • Supported file types: .pdf, .docx, .pptx
  • Max file size: 10 MB per file
  • Add metadata: title, category (subject/level), description, upload date
  • Optional: delete or update PQs

Public User Functionality:

  • Browse PQs by subject/category
  • Download PQ files directly
  • View basic metadata: title, category, description, upload date

Backend Specifications:

  • File storage: secure server or cloud storage (S3)

API endpoints:

  • POST /pq/upload – Admin upload
  • GET /pq/list – Public listing
  • GET /pq/download/:id – Public download
  • DELETE /pq/:id – Admin delete/update

Additional Features:

  • Ability for Users to search a pq
  • Ability to filter by department, course
  • Pagination

UI Specifications:

  • Admin Upload Page: form with title, category dropdown, description textarea, file selector, submit button
  • Public PQ Listing Page: table or card layout with PQs, download buttons per PQ
  • Search/filter by category or title
  • Pagination

Non-Functional Requirements:

  • File validation (type and size)
  • Secure storage and download links
  • Fast retrieval and download
  • Accessible layout for public users

4. Backend Specifications

  • Technology Stack: PHP (Laravel) or Node.js (Express) for backend, MySQL/PostgreSQL database
  • API Endpoints:
    • GET /news, /articles, /events, /p
    • POST /frms/report
    • GET /frms/track/:id
    • Admin APIs: GET /frms/all, PUT /frms/update, POST /frms/assign
  • Authentication: Only for admin/staff (JWT or session-based)
  • File Storage: Uploaded images/files for FRMS stored securely
  • Data Validation: Backend validation of all forms
  • Notifications: Optional email notifications for staff when fault is submitted or status updated

Database Tables: - News: id, title, content, date, category, author - Events: id, title, description, date, location - Articles: id, title, content, author, date, category - FRMS: id, name, email, phone, location, category, description, file, reference_id, status, assigned_staff, created_at, updated_at - Staff/Admin Users: id, name, email, role, password, created_at, updated_at - Categories (for FRMS): id, name, description

5. UI / Design Specifications

  • Design Style: Formal, academic, credible, minimal, professional
  • Color Palette: White background, deep navy primary (#0B2C4D), subtle gray dividers
  • Typography: Serif headings (Merriweather), Sans-serif body (Inter/Open Sans)
  • Layout: Structured grid, two-column sections for content-heavy pages
  • Navigation: Top horizontal navigation bar, footer with institutional info
  • Forms & Modals: Report Fault, Track Fault, Event Registration popup - all modals redesign full page including dimmed background
  • Responsive Design: Desktop-first, mobile-friendly with hamburger menu for sidebar when needed
  • Accessibility: Proper color contrast, readable fonts, keyboard navigation

Key CTAs: - Report a Fault - Track My Fault - View Event Details - Read Articles/News - Submit Contact Form

6. Non-Functional Requirements

  • Security: HTTPS, form validation, secure file uploads
  • Performance: Page load <2s, efficient DB queries
  • Scalability: Support growing number of events/news and FRMS submissions
  • Backup & Restore: Daily DB backups
  • Accessibility: WCAG 2.1 compliance
  • SEO: Proper headings, meta tags, semantic HTML

7. Deliverables

  • Fully functional public-facing website with all pages listed
  • FRMS system for public reporting and admin management
  • Backend APIs for content and FRMS management
  • UI design consistent with specifications, ready for implementation
  • Database schema for all content and FRMS tables
